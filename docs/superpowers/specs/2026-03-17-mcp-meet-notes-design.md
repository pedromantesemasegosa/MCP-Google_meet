# MCP Meet Notes — Design Spec

## Purpose

An MCP Server that syncs Google Meet transcriptions and summaries (generated by Google Gemini) from Google Drive to local storage, and exposes tools for searching, cross-referencing, and tracking topic progression across meetings — all from Claude Code or Cursor via natural conversation.

## Requirements

- Daily automatic sync of Gemini meeting notes from Google Drive to local Markdown files
- MCP Server with 6 tools for querying meetings conversationally
- Cross-meeting intelligence: relate information across calls and track topic evolution
- Google Workspace account support with full setup guide from scratch
- Local-first: all data stored on disk for fast access and portability

## Architecture

Three independent components connected through the filesystem:

```
Google Drive (Gemini notes)
        │
        ▼
┌─────────────────────────────┐
│   SYNCER (sync_meetings.py) │  ← runs daily via launchd
│   - OAuth2 → Drive API      │
│   - Detects new/modified    │
│   - Converts to Markdown    │
│   - Updates index.json      │
└──────────┬──────────────────┘
           │ writes
           ▼
┌─────────────────────────────┐
│   STORAGE (meetings/)       │
│   - index.json (registry)   │
│   - YYYY-MM-DD-title.md     │
└──────────┬──────────────────┘
           │ reads
           ▼
┌─────────────────────────────┐
│   MCP SERVER (server.py)    │  ← connects to Claude Code / Cursor
│   - 6 query tools           │
│   - Filters via index.json  │
│   - Reads .md for content   │
└─────────────────────────────┘
```

The Syncer writes; the MCP Server reads. No runtime dependency between them beyond the filesystem. The Syncer can run while the MCP Server is active without conflicts.

## Data Model

### Meeting file (Markdown with YAML frontmatter)

```markdown
---
id: "abc123"
title: "Standup Equipo Backend"
date: "2026-03-17T10:00:00+01:00"
duration_minutes: 25
participants:
  - "pedro.mantese@feverup.com"
  - "laura.garcia@feverup.com"
  - "carlos.ruiz@feverup.com"
source_doc_id: "1aBcDeFgHiJkLmNoPqRsTuVwXyZ"
synced_at: "2026-03-17T12:00:00+01:00"
tags: []
---

## Resumen

Gemini's summary goes here...

## Transcripcion

Full transcript from Gemini...

## Action Items

- [ ] Pedro: Revisar PR de migracion
- [ ] Laura: Preparar demo para el viernes
```

### index.json

```json
{
  "last_sync": "2026-03-17T12:00:00+01:00",
  "meetings": [
    {
      "id": "abc123",
      "title": "Standup Equipo Backend",
      "date": "2026-03-17T10:00:00+01:00",
      "duration_minutes": 25,
      "participants": [
        "pedro.mantese@feverup.com",
        "laura.garcia@feverup.com",
        "carlos.ruiz@feverup.com"
      ],
      "file": "2026-03-17-standup-equipo-backend.md",
      "source_doc_id": "1aBcDeFgHiJkLmNoPqRsTuVwXyZ",
      "has_action_items": true,
      "summary": "Revision del progreso de migracion a K8s, bloqueo en CI/CD por permisos, y preparacion de demo para stakeholders del viernes."
    }
  ]
}
```

The `summary` field provides a 1-2 line overview of topics discussed, enabling differentiation between similarly-titled meetings without reading the full file.

### Storage layout

Flat directory. All meetings in `meetings/` ordered by filename (date prefix). The index.json handles logical organization and relationships.

```
meetings/
├── index.json
├── 2026-03-15-planning-sprint-42.md
├── 2026-03-16-1on1-laura.md
├── 2026-03-17-standup-equipo-backend.md
└── ...
```

## MCP Server Tools

### 1. `search_by_topic(query: str, limit?: int) → list[Meeting]`

Searches meetings related to a topic. First searches index.json (titles + summaries). If more depth is needed, searches inside .md content. Returns matching meetings with their summaries.

### 2. `search_by_participant(participant: str, date_from?: str, date_to?: str) → list[Meeting]`

Filters meetings by participant. Supports partial matching (e.g., "laura" matches "laura.garcia@feverup.com"). Optional date range filter.

### 3. `search_by_date(date_from: str, date_to?: str) → list[Meeting]`

Returns meetings in a date range. If only `date_from` is provided, returns meetings from that single day.

### 4. `get_topic_progression(topic: str) → list[MeetingWithContent]`

Finds all meetings mentioning a topic, orders them chronologically, and returns summary + relevant excerpts from each. Designed for the LLM to interpret the evolution of a topic over time.

### 5. `get_action_items(date_from?: str, date_to?: str, participant?: str) → list[ActionItem]`

Extracts action items from filtered meetings. Can filter by date range and/or responsible person.

### 6. `get_executive_summary(date_from: str, date_to?: str) → list[MeetingSummary]`

Returns titles, summaries, and action items for all meetings in a range. Designed for "give me a summary of this week."

All tools return structured JSON. The LLM interprets and presents the data conversationally.

## Syncer

### Authentication

- OAuth2 with persistent token at `config/token.json`
- First-time setup requires interactive browser login via `scripts/first_auth.py`
- Token auto-refreshes. If refresh token expires (~6 months on Workspace), logs a warning and requires manual re-auth.

### Sync process

1. Read `last_sync` from `index.json`
2. Query Google Drive API for documents in the Gemini notes folder modified since `last_sync`
3. For each new/modified document:
   a. Download content via Drive API (export as plain text)
   b. Extract metadata: title, date, participants, duration
   c. Generate short summary for the index
   d. Convert to Markdown with YAML frontmatter
   e. Name file: `YYYY-MM-DD-titulo-slugificado.md`
   f. Add/update entry in `index.json`
4. Update `last_sync` timestamp

### Idempotency

Uses `source_doc_id` to detect already-synced documents. If a Drive document was modified after sync, the local .md is updated rather than duplicated.

### Scheduling

macOS `launchd` plist in `~/Library/LaunchAgents/` running once daily. Includes:
- `scripts/install_launchd.sh` — places and activates the plist
- The plist file itself, ready to install

## Error Handling

| Scenario | Behavior |
|----------|----------|
| No internet connection | Syncer fails silently, logs error, retries next cycle. MCP continues serving local data. |
| OAuth token expired | Refresh token renews automatically. If refresh token expires, logs warning requiring manual re-auth. |
| Unexpected document format | Parses what it can, saves document with `parse_warnings` flag in frontmatter. |
| Corrupted index.json | Auto-regenerates by scanning existing .md files and reading their frontmatters. |
| Duplicate meeting | Detected by `source_doc_id`, updates existing file instead of creating duplicate. |

## Logging

Syncer logs to `logs/sync.log` with daily rotation, keeping the last 30 days. Captures: sync start/end, documents found, documents processed, errors encountered.

## Configuration

`config/settings.json`:

```json
{
  "meetings_dir": "meetings",
  "drive_folder_name": "Meeting notes",
  "sync_interval_description": "daily via launchd"
}
```

## Tech Stack

- **Language**: Python 3.11+
- **MCP SDK**: `mcp` (Python SDK)
- **Google APIs**: `google-api-python-client`, `google-auth-oauthlib`
- **Scheduling**: macOS `launchd`
- **No database**: filesystem only (Markdown + JSON)

## Out of Scope

- Semantic search / embeddings (can be added later if needed)
- Manual sync trigger tool (user explicitly excluded)
- Web UI or any interface beyond MCP tools
- Automatic topic tagging (tags field exists but starts empty)
- Real-time sync (daily is sufficient)
