# MCP Meet Notes — Design Spec

## Design Summary

```yaml
design_type: new_feature
risk_level: medium
complexity_level: medium
complexity_rationale: >
  Three independent components (syncer, storage, MCP server) with external
  dependency on Google Drive API and OAuth2 token lifecycle. Gemini's note
  format is undocumented and may change. Filesystem coordination requires
  atomic writes.
main_constraints:
  - Google Workspace may restrict OAuth scopes or Drive API access
  - Gemini note format is not formally documented by Google
  - macOS-only scheduling (launchd)
biggest_risks:
  - Gemini changes its meeting notes output format without notice
  - OAuth refresh token expiration on Workspace accounts (~6 months)
  - Large meeting archives (1000+ files) degrading full-text search performance
unknowns:
  - Exact Gemini notes structure across different meeting types (1:1 vs group)
  - Whether Workspace admin policies block Drive API OAuth for desktop apps
```

## Purpose

An MCP Server that syncs Google Meet transcriptions and summaries (generated by Google Gemini) from Google Drive to local storage, and exposes tools for searching, cross-referencing, and tracking topic progression across meetings — all from Claude Code or Cursor via natural conversation.

## Requirements

- Daily automatic sync of Gemini meeting notes from Google Drive to local Markdown files
- MCP Server with 7 tools for querying meetings conversationally
- Cross-meeting intelligence: relate information across calls and track topic evolution
- Google Workspace account support with full setup guide from scratch
- Local-first: all data stored on disk for fast access and portability

## Architecture

Three independent components connected through the filesystem:

```
Google Drive (Gemini notes)
        │
        ▼
┌─────────────────────────────┐
│   SYNCER (sync_meetings.py) │  ← runs daily via launchd
│   - OAuth2 → Drive API      │
│   - Detects new/modified    │
│   - Converts to Markdown    │
│   - Updates index.json      │
└──────────┬──────────────────┘
           │ writes
           ▼
┌─────────────────────────────┐
│   STORAGE (meetings/)       │
│   - index.json (registry)   │
│   - YYYY-MM-DD-title.md     │
└──────────┬──────────────────┘
           │ reads
           ▼
┌─────────────────────────────┐
│   MCP SERVER (server.py)    │  ← connects to Claude Code / Cursor
│   - 7 query tools           │
│   - Filters via index.json  │
│   - Reads .md for content   │
└─────────────────────────────┘
```

The Syncer writes; the MCP Server reads. No runtime dependency between them beyond the filesystem. The Syncer can run while the MCP Server is active without conflicts.

## Data Model

### Meeting file (Markdown with YAML frontmatter)

```markdown
---
id: "abc123"
title: "Standup Equipo Backend"
date: "2026-03-17T10:00:00+01:00"
duration_minutes: 25
participants:
  - "pedro.mantese@feverup.com"
  - "laura.garcia@feverup.com"
  - "carlos.ruiz@feverup.com"
source_doc_id: "1aBcDeFgHiJkLmNoPqRsTuVwXyZ"
synced_at: "2026-03-17T12:00:00+01:00"
tags: []
---

## Resumen

Gemini's summary goes here...

## Transcripcion

Full transcript from Gemini...

## Action Items

- [ ] Pedro: Revisar PR de migracion
- [ ] Laura: Preparar demo para el viernes
```

### index.json

```json
{
  "last_sync": "2026-03-17T12:00:00+01:00",
  "meetings": [
    {
      "id": "abc123",
      "title": "Standup Equipo Backend",
      "date": "2026-03-17T10:00:00+01:00",
      "duration_minutes": 25,
      "participants": [
        "pedro.mantese@feverup.com",
        "laura.garcia@feverup.com",
        "carlos.ruiz@feverup.com"
      ],
      "file": "2026-03-17-standup-equipo-backend.md",
      "source_doc_id": "1aBcDeFgHiJkLmNoPqRsTuVwXyZ",
      "has_action_items": true,
      "summary": "Revision del progreso de migracion a K8s, bloqueo en CI/CD por permisos, y preparacion de demo para stakeholders del viernes."
    }
  ]
}
```

The `summary` field provides a 1-2 line overview of topics discussed, enabling differentiation between similarly-titled meetings without reading the full file.

### Storage layout

Flat directory. All meetings in `meetings/` ordered by filename (date prefix). The index.json handles logical organization and relationships.

```
meetings/
├── index.json
├── 2026-03-15-planning-sprint-42.md
├── 2026-03-16-1on1-laura.md
├── 2026-03-17-standup-equipo-backend.md
└── ...
```

## MCP Server Tools

### 1. `search_by_topic(query: str, limit?: int) → list[Meeting]`

Searches meetings related to a topic using a two-phase strategy:
1. **Index search**: scans `index.json` titles and summaries for keyword matches.
2. **Full-text fallback**: if fewer than `limit` results are found (default `limit=5`), falls back to searching inside `.md` file content for additional matches.

Returns matching meetings with their summaries, ordered by relevance (title match > summary match > body match).

### 2. `search_by_participant(participant: str, date_from?: str, date_to?: str) → list[Meeting]`

Filters meetings by participant. Supports partial matching (e.g., "laura" matches "laura.garcia@feverup.com"). Optional date range filter.

### 3. `search_by_date(date_from: str, date_to?: str) → list[Meeting]`

Returns meetings in a date range. If only `date_from` is provided, returns meetings from that single day.

### 4. `get_topic_progression(topic: str) → list[MeetingWithContent]`

Finds all meetings mentioning a topic, orders them chronologically, and returns summary + relevant excerpts from each. Designed for the LLM to interpret the evolution of a topic over time.

### 5. `get_action_items(date_from?: str, date_to?: str, participant?: str) → list[ActionItem]`

Extracts action items from filtered meetings. Can filter by date range and/or responsible person.

### 6. `get_executive_summary(date_from: str, date_to?: str) → list[MeetingSummary]`

Returns titles, summaries, and action items for all meetings in a range. Designed for "give me a summary of this week."

### 7. `get_sync_status() → SyncStatus`

Returns the current state of the sync system: last successful sync timestamp, total meetings indexed, and the result of the most recent run (success/failed/unknown) parsed from `logs/sync.log`. Useful for diagnosing sync issues without leaving the chat.

All tools return structured JSON. The LLM interprets and presents the data conversationally.

## Syncer

### Authentication

- OAuth2 with persistent token at `config/token.json`
- First-time setup requires interactive browser login via `scripts/first_auth.py`
- Token auto-refreshes. If refresh token expires (~6 months on Workspace), logs a warning and requires manual re-auth.

### Expected Gemini notes format

Google Gemini generates meeting notes as Google Docs with a consistent structure:

```
Meeting title (as document title)
Date: March 17, 2026
Duration: 25 minutes
Attendees: Pedro Mantese, Laura Garcia, Carlos Ruiz

Summary
[Gemini-generated summary paragraph]

Transcript
[Speaker-attributed transcript]

Action items
- Pedro: Review migration PR
- Laura: Prepare Friday demo
```

The `document_parser.py` extracts metadata using heading-based parsing:
- **Title**: from the Google Doc title (Drive API metadata)
- **Date**: parsed from the "Date:" line or the document creation date as fallback
- **Participants**: parsed from the "Attendees:" line. On first sync or when no prior meetings exist, stores display names as-is (e.g., "Pedro Mantese"). Optionally, a `config/known_participants.json` mapping file can provide name-to-email resolution.
- **Duration**: parsed from the "Duration:" line if present, otherwise `null`
- **Summary for index**: first 1-2 sentences from the "Summary" section
- **Action items detection**: presence of an "Action items" section with list items sets `has_action_items: true`

**Heading translation**: Gemini outputs in English (Summary, Transcript, Action items). The syncer converts these to Spanish headings in the local Markdown files (`## Resumen`, `## Transcripcion`, `## Action Items`). The parser recognizes BOTH English and Spanish headings to handle edge cases and index regeneration.

If Gemini changes its output format, the parser logs a warning and falls back to: title from Drive metadata, date from document creation date, full document body as transcript, no structured action items.

### Sync process

1. Read `last_sync` from `index.json`
2. Query Google Drive API for documents in the Gemini notes folder modified since `last_sync`
3. For each new/modified document:
   a. Download content via Drive API (export as plain text)
   b. Parse document using heading-based extraction (see format above)
   c. Generate short summary from Gemini's own summary section
   d. Convert to Markdown with YAML frontmatter
   e. Write file atomically (write to `.tmp` then rename) to prevent partial reads
   f. Name file: `YYYY-MM-DD-titulo-slugificado.md`
   g. Add/update entry in `index.json` (also written atomically)
4. Update `last_sync` timestamp

### Idempotency

Uses `source_doc_id` to detect already-synced documents. If a Drive document was modified after sync, the local .md is updated rather than duplicated.

### Scheduling

macOS `launchd` plist in `~/Library/LaunchAgents/` running once daily at 9:00 AM. Includes:
- `scripts/install_launchd.sh` — places and activates the plist
- The plist file itself, ready to install

On transient failures (no network at startup, OAuth refresh error), the syncer retries up to **3 times with a 60-second delay** between attempts before giving up. This handles the common case of the machine having no network immediately after waking from sleep.

## Error Handling

| Scenario | Behavior |
|----------|----------|
| No internet connection | Syncer retries up to 3 times (60s between attempts), logs each failure. If all attempts fail, logs final error and exits. MCP continues serving local data. |
| OAuth token expired | Refresh token renews automatically. If refresh token expires, logs warning requiring manual re-auth. |
| Unexpected document format | Parses what it can, saves document with `parse_warnings` flag in frontmatter. |
| Corrupted index.json | Auto-regenerates by scanning existing .md files: reads YAML frontmatter for metadata, parses Markdown body to reconstruct `has_action_items` (scans for `## Action Items` section with checklist items) and `summary` (extracts first 1-2 sentences from `## Resumen` section). |
| Duplicate meeting | Detected by `source_doc_id`, updates existing file instead of creating duplicate. |

## Logging

Syncer logs to `logs/sync.log` with daily rotation, keeping the last 30 days. Captures: sync start/end, documents found, documents processed, errors encountered.

## Tech Stack

- **Language**: Python 3.11+
- **MCP SDK**: `fastmcp` — high-level wrapper over the official MCP Python SDK. Simplifies tool definition via decorators (`@mcp.tool()`) instead of manual protocol boilerplate.
- **Google APIs**: `google-api-python-client`, `google-auth-oauthlib`
- **Scheduling**: macOS `launchd`
- **No database**: filesystem only (Markdown + JSON)

## Acceptance Criteria

1. **When** the syncer runs and new Gemini notes exist in Drive, **the system shall** download them, convert to Markdown with YAML frontmatter, and add entries to `index.json`.
2. **When** a previously synced document has been modified in Drive, **the system shall** update the local `.md` file and its `index.json` entry rather than creating a duplicate.
3. **If** `index.json` is corrupted or missing, **then** the system shall regenerate it by scanning existing `.md` files (frontmatter for metadata, body for `has_action_items` and `summary`).
4. **When** `search_by_topic` returns fewer than `limit` results from the index, **the system shall** fall back to full-text search of `.md` files.
5. **When** `search_by_participant` receives a partial name (e.g., "laura"), **the system shall** match it against full participant identifiers (e.g., "laura.garcia@feverup.com").
6. **When** `get_topic_progression` is called, **the system shall** return matching meetings ordered chronologically with summaries and relevant excerpts.
7. **When** `get_action_items` is called with a participant filter, **the system shall** return only action items assigned to that person.
8. **If** a Gemini document has an unexpected format, **then** the system shall parse what it can, save the document with a `parse_warnings` field in frontmatter, and log a warning.
9. **While** the syncer is writing files, **the system shall** use atomic writes (temp file + rename) to prevent partial reads by the MCP server.
10. **When** the syncer has no internet connection, **the system shall** log the error and exit gracefully; the MCP server shall continue serving existing local data.

## Test Strategy

### Unit tests
- **`document_parser.py`**: Parse standard Gemini format, handle missing sections, handle unexpected format (fallback behavior), extract action items, generate summary.
- **`search.py`**: Two-phase search (index-only, fallback to full-text), partial participant matching, date range filtering, topic progression ordering.
- **`index_manager.py`**: Add/update/read entries, regeneration from `.md` files, atomic write behavior.

### Integration tests (mocked Drive API)
- **Syncer end-to-end**: Mock Google Drive API responses, verify correct `.md` files and `index.json` entries are created. Test idempotency (re-sync same document). Test modified document update.
- **MCP Server tools**: Load a test `meetings/` directory with known data, verify each tool returns correct results.

### E2E test
- Manual verification: run syncer against real Google Drive, confirm files are created, then query via MCP tools in Claude Code.

## Security Considerations

- **OAuth scope**: Request minimum required scope (`https://www.googleapis.com/auth/drive.readonly`) — read-only access to Drive.
- **Credentials storage**: `credentials.json` and `token.json` are gitignored and must never be committed. `.gitignore` enforced from project creation.
- **Local data sensitivity**: Meeting transcriptions are stored unencrypted on local disk. Users should be aware these files contain potentially sensitive conversation content.
- **Log hygiene**: `sync.log` must never contain meeting content, tokens, or credentials — only operational metadata (timestamps, document IDs, error types).

## Out of Scope

- Semantic search / embeddings (can be added later if needed)
- Manual sync trigger tool (user explicitly excluded)
- Web UI or any interface beyond MCP tools
- Automatic topic tagging (tags field exists but starts empty)
- Real-time sync (daily is sufficient)
